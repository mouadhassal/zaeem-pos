# T1.0a — Command Inventory: Frontend `getDb()` calls → proposed Rust commands

Generated as part of the S1 / multi-tenant migration (see `ARCHITECTURE_V3.md`). This is a
research artifact only — no application code was changed to produce it.

**Method:** every one of the 26 files listed in the task was read in full and every
`.selectFrom(...)` / `.insertInto(...)` / `.updateTable(...)` / `.deleteFrom(...)` / raw `sql`
chained off a `getDb()` call was cataloged. Related CRUD/queries that fire together for one screen
action or one screen load are grouped under a single proposed Rust command where there's a natural
transactional or UI boundary; groupings are called out explicitly in each row.

**Roles in the current schema:** `CASHIER`, `MANAGER`, `ADMIN`, `OWNER`, `ACCOUNTANT`, `KITCHEN`.
`ADMIN` is treated identically to `MANAGER` everywhere in current frontend gating
(`src/lib/permissions.ts`). None of these map 1:1 onto ARCHITECTURE_V3.md's `Platform` role — that
role does not exist as a UI-reachable concept in the current single-tenant app.

**Scope column:** since the current DB has no `tenant_id`/`branch_id` columns anywhere, "Scope" is
an *inference* of what each operation will become once multi-tenancy lands, per ARCHITECTURE_V3.md
§2/§3. Almost everything is Branch-scoped (this is a single-restaurant install). Loyalty
(`customers`, `loyalty_cards`, `loyalty_transactions`) is Tenant-scoped per §6. Chain-wide
`chain_config` (tax/currency/branding defaults) is Tenant-scoped per §2 "Set tenant-default
menu/prices." Branch creation/licensing is Platform-scoped per hard rule #1. AI chat is
Owner/Platform-only per §2, and its aggregate queries are treated as Tenant-scoped.

- Internal duplication (not vs. existing Rust, but within the frontend itself): the manager-PIN
  verification logic in `VoidItemModal.tsx` and `ManagerPinModal.tsx` are two divergent
  implementations of the same check (one has lockout/failure tracking, one doesn't) and must be
  unified into a single `verify_manager_pin` Rust command.

### Ambiguous items needing a human decision

**Hard-rule-adjacent (branches/tenants/staff/licensing) — highest priority:**
1. **`create_branch` is called from TWO places today** (`app/branches/page.tsx:236-239` and
   `app/settings/page.tsx:199-215`) and both are reachable by roles other than Platform — one by
   OWNER (`canAccessBranches`), one silently by MANAGER/ADMIN (`canAccessSettings`, via the
   `else` branch of `saveBranch()` when no branch row exists yet). Per hard rule #1 this must be
   rejected server-side for any non-Platform caller. This is the single most direct violation of
   ARCHITECTURE_V3.md §2 found in the inventory — needs explicit confirmation that both call sites
   collapse to one Platform-only Rust command and that the frontend removes the OWNER/MANAGER-facing
   UI entry points (or repoints them at a "request a new branch from platform" flow instead).
2. **`update_branch` role boundary** — is MANAGER allowed to edit their own branch's `name`/
   `address`/`city`, or only operational fields like `max_tables`? §2 doesn't explicitly enumerate
   branch-info-editing as distinct from "override menu/price locally"; current inference (Branch
   scope, Manager allowed for own branch) is an analogy, not a direct match.
3. **`set_branch_active`** (enable/disable a branch) — not explicitly enumerated in §2; inferred
   Owner/Platform-only (Tenant scope) since disabling a branch is a tenant-wide business decision,
   but needs confirmation it shouldn't be Manager-accessible.
4. **`create_staff`/`update_staff` role field** — the UI lets the caller assign *any* role
   (including OWNER/MANAGER) to a new or edited staff member with no check based on the caller's
   own role. Hard rule #2 only specifies branch_id forcing; it's silent on whether a MANAGER should
   be blocked from promoting someone to MANAGER or OWNER. Needs an explicit policy decision.
5. **`force_close_shift`** — frontend gates the button to literal `role === "MANAGER"`, deliberately
   excluding ADMIN and OWNER even though both normally have equal-or-greater privilege elsewhere.
   Unclear if intentional or an oversight; final Permission column needs a human call.
6. **`clock_in`/`clock_out`** — no role check beyond page-level `canAccessStaff`; UI implies a
   manager clocking *other* staff in/out rather than self-service. If cashiers/kitchen staff are
   meant to clock themselves in elsewhere in the app, this command's permission set needs to expand.
7. **Chain-wide settings (`update_settings` currency/tax)** — per §2 only Platform/Owner may "set
   tenant-default…prices," but `canAccessSettings` currently admits MANAGER/ADMIN with no
   additional check. This is a real gap between current UI and the target model that the Rust
   command's permission check must close (and likely needs a genuinely separate branch-level
   override table/command for Manager, which doesn't exist today).
8. **`get_printers`/`update_printer` scope** — classified Branch (a printer is physical hardware at
   one location), but there is no `branch_id` column on `printers` at all yet; a schema migration
   is a prerequisite, not just a Rust command change.
9. **`get_branches_overview`/`get_branch_detail` scoping bugs** — the staff-count and today-sales
   sub-queries in `branches/page.tsx` have no `branch_id` WHERE clause today and silently return
   global figures mislabeled as per-branch stats. Same bug class affects `get_staff_list`,
   `get_shifts`, and `get_attendance` in `staff/page.tsx` — none of these are scoped today (expected,
   pre-migration), but they are the highest-priority queries to scope correctly once `branch_id`
   exists, since a leak here directly violates hard rule #3.

**Lower-priority / data-model ambiguity:**
10. `customers/page.tsx` joins `orders`↔`customers` via `customer_phone`, not a stable
    `customer_id` — editing a customer's phone would orphan their order history under the current
    join key; worth a schema fix alongside the Rust migration.
11. `shift/page.tsx`'s manager-override check (`verify_manager_override`) currently fetches raw
    password/PIN hashes into the renderer and verifies client-side — a real security bug (not just
    an architecture-migration nicety) that must not be replicated in the new command; the hash
    should never leave Rust.
12. `debug/page.tsx` — should its backing commands (`debug_get_db_stats`) exist in release builds
    at all, or should the whole page and its commands be compiled out via `#[cfg(debug_assertions)]`
    the way `seed_default_users` already is in `lib.rs`? Recommend the latter; flagging since it's a
    scope-model question (Platform vs. "doesn't exist").
