# T2.0 — Supplier Ledger, Per-Terminal Licensing, Owner Dashboard, Loyalty Rethink

**Status: PLAN-FIRST — review gate. No migration code, no Rust, no frontend code written against this yet.**
**Authority chain:** `AGENTS.md` (prime directives, R1-R5) → `ARCHITECTURE_V3.md` (multi-tenant hierarchy, §7 cloud scope) → `SCHEMA_V3.md` (money model, sync shapes, current target schema).

This document covers four independently-scoped systems requested together. Each section is:
schema changes → approach → what it touches → effort estimate. A "Conflicts & flags" section
sits up front because two of the flags apply to all four, and one is a documentation correction
that should happen regardless of whether any of this gets built.

## 0. Conflicts & flags (read first)

1. **AGENTS.md prime directive #2: "Do not add features. The feature surface is frozen... If a
   task seems to require a new feature, stop and ask."** All four items below are new features.
   This document IS the "stop and ask" — it designs and stops for review, it does not build.
   Flagging explicitly: approving this plan means formally lifting the frozen-surface rule for
   this scope, not quietly working around it.
2. **`SCHEMA_V3.md` §9.1's claim is false and should be corrected independently of this plan.**
   It says the local `terminal` table is "aligned with `zaeem-control`'s existing Postgres
   `Terminal` model." Direct exploration of `apps/control` found no Prisma model at all —
   `apps/control` talks to Supabase via raw SQL / service-role client, no ORM. That line in
   `SCHEMA_V3.md` is aspirational, not a description of anything that exists. §2 below actually
   builds the real thing that line was describing.
3. **Money-column convention is inconsistent in the current schema**, and new code must pick the
   correct side of that inconsistency. `debt_entries.amount` got the full `MoneySnapshot` column
   expansion (`_minor/_currency/_scale/_base_minor/_fx_rate/_fx_source/_denom_epoch`) during
   Migration A; `purchase_orders.total_cents` and `purchase_order_items.unit_cost_cents` did not
   — they're still bare legacy `_cents` integers. Per AGENTS.md R3 ("a raw `i64` labelled
   `_cents` is banned in any new code"), every new money column proposed below uses the full
   `MoneySnapshot` shape, even where it sits next to an older bare-`_cents` column it references.
4. **Cross-branch aggregation (needed by §3, the owner dashboard) requires the cloud.**
   `ARCHITECTURE_V3.md` §5/§7 is explicit: local SQLite is the source of truth for *one branch's*
   sales; the cloud aggregates, and POS reads/writes must never move to the cloud. Today's sync
   set (`sync_outbox`, per `sync.rs`) carries **only `orders`/`order_items`/`payments`** —
   `operational_costs`, `suppliers`, `purchase_orders` are not synced at all. A multi-branch
   owner cannot see branch-vs-branch cost/profit comparison until that sync set is widened. This
   is a real, load-bearing dependency of §3 on both §1 (new supplier ledger table to sync) and on
   a currently-just-missing sync wire-up for `operational_costs`.
5. **A real integrity gap exists in today's loyalty system**, found during exploration, unrelated
   to any prior task: `earn_loyalty_points_v3` trusts a caller-supplied point amount rather than
   computing it server-side, and the POS wraps the earn call in a bare `try {} catch {}` that
   silently drops points on failure. §4 below fixes this as a natural consequence of moving
   accrual into the server-side payment transaction — flagged here so it doesn't read as scope
   creep when it shows up in that section.

## 1. Supplier ledger

### Current state (grounding facts)
- `suppliers(id, name, phone, email, total_orders, total_purchases_cents, is_active, ...)` — no
  balance/payment tracking of any kind today.
- `purchase_orders(id, supplier_id, status, total_cents, created_at, received_at, ...)` — no
  `payment_status`/`amount_paid` field.
- `receive_purchase_order` (`repo.rs:2056`) only touches inventory: bumps
  `ingredients.current_stock`, writes an `inventory_logs` row, flips `status → RECEIVED`. It
  **never touches money.** `operational_costs` is a fully separate, manually-entered table with
  zero automatic linkage to purchase orders.
- Multi-tenant scoping (`tenant_id`/`branch_id` + `Scope`-checked repo calls) is **already live**
  for `suppliers`/`purchase_orders`/`purchase_order_items` — this is infrastructure, not a gap.
- The `debtors`/`debt_entries` pair is the exact right precedent: a cached running-balance row
  (`debtors.total_debt_cents/total_paid_cents/balance_cents`) plus an append-only fact table
  (`debt_entries`, `type IN ('DEBT','PAYMENT')`) that the balance is a rollup of. `repo.rs`'s
  `record_debt_payment` is the pattern to mirror line-for-line.

### Schema

```sql
-- running balance cache on suppliers, mirrors debtors' three columns exactly
ALTER TABLE suppliers ADD COLUMN total_owed_minor  INTEGER NOT NULL DEFAULT 0;
ALTER TABLE suppliers ADD COLUMN total_paid_minor  INTEGER NOT NULL DEFAULT 0;
ALTER TABLE suppliers ADD COLUMN balance_minor     INTEGER NOT NULL DEFAULT 0;  -- owed - paid; negative = credit/advance held

-- append-only fact table, mirrors debt_entries
CREATE TABLE supplier_payments (
  id                 TEXT PRIMARY KEY,        -- UUIDv7
  tenant_id          TEXT NOT NULL, branch_id TEXT NOT NULL,
  supplier_id        TEXT NOT NULL REFERENCES suppliers(id),
  purchase_order_id  TEXT REFERENCES purchase_orders(id),   -- NULL = standalone payment (e.g. settling an old invoice), not tied to one receive event
  type               TEXT NOT NULL CHECK(type IN ('CHARGE','PAYMENT')),  -- CHARGE = money now owed (PO received); PAYMENT = money paid out
  amount_minor       INTEGER NOT NULL, amount_currency TEXT NOT NULL, amount_scale INTEGER NOT NULL,
  amount_base_minor  INTEGER NOT NULL, amount_fx_rate TEXT NOT NULL, amount_fx_source TEXT NOT NULL,
  amount_denom_epoch INTEGER NOT NULL DEFAULT 2,
  method             TEXT,                     -- 'CASH'|'BANK'|'CARD', nullable for CHARGE rows
  notes              TEXT,
  created_by         TEXT NOT NULL REFERENCES staff(id),
  created_at         TEXT NOT NULL DEFAULT (datetime('now')),
  device_id TEXT NOT NULL, seq INTEGER NOT NULL   -- standard fact-row sync shape per SCHEMA_V3 §6
);

-- traceability: let an auto-generated operational_cost row point back at what created it,
-- so it's never manually double-entered and is filterable/excludable in reports
ALTER TABLE operational_costs ADD COLUMN reference_type TEXT;   -- e.g. 'supplier_payment'
ALTER TABLE operational_costs ADD COLUMN reference_id   TEXT;

-- denormalized cache for the PO list screen (avoid a join on every row render)
ALTER TABLE purchase_orders ADD COLUMN amount_paid_minor INTEGER NOT NULL DEFAULT 0;
ALTER TABLE purchase_orders ADD COLUMN payment_status TEXT NOT NULL DEFAULT 'UNPAID'
  CHECK(payment_status IN ('UNPAID','PARTIAL','PAID','ADVANCE'));
```

### Approach

- **Receive flow, extended, not replaced.** The existing receive screen (quantities per item)
  gains one section: "المبلغ المدفوع الآن" (amount paid now), defaulting to 0, with quick actions
  for "دفع كامل" (pay in full) / leave at 0 for unpaid. `receive_purchase_order_v3` gains an
  optional `amount_paid_minor` argument. In the **same transaction** as today's inventory-receive
  logic: (a) insert a `supplier_payments` CHARGE row for `total_cents` (the PO now owes this
  much), (b) if `amount_paid_minor > 0`, insert a `supplier_payments` PAYMENT row, (c) update
  `suppliers.total_owed_minor/total_paid_minor/balance_minor` and
  `purchase_orders.amount_paid_minor/payment_status` from those two facts, (d) if a PAYMENT was
  recorded, auto-insert one `operational_costs` row (category "مشتريات من الموردين",
  `reference_type='supplier_payment'`, `reference_id = <the new row's id>`) so the existing
  Finance → Costs tab picks it up **for free**, with no new query logic — this is the mechanism
  that satisfies "money out to suppliers must appear in the owner's cash-flow picture" without
  building a second, parallel bookkeeping surface.
- **Standalone payments** (settling last month's invoice, not tied to a fresh receive): a new
  "الموردون" ledger page, built by copying `debt/page.tsx`'s structure almost verbatim — supplier
  list with owed/paid/balance columns, click-through to a detail drawer showing PO + payment
  history, a "تسديد دفعة" button that calls a new `record_supplier_payment_v3` command
  (mirrors `record_debt_payment_v3` exactly: looks up the supplier's own `tenant_id`/`branch_id`
  from the row via `Scope`, so an Owner in `Scope::Tenant` can still act, same as the debt case).
- **Advance payments**: modeled as an ordinary `PAYMENT` row with no matching `CHARGE` yet —
  `balance_minor` goes negative, which the UI shows as a credit ("رصيد مسبق: ٥٠٠"). The next PO
  received from that supplier nets against it automatically since the balance is a running sum,
  not a per-PO ledger — no special-case code needed for "advance" as a type.
- **Audit**: every `supplier_payments` insert writes one `audit_log` entry in the same
  transaction (`Action::SupplierPaymentRecorded`), same shape as `record_debt_payment`'s existing
  audit call — this is money leaving the business, same rigor as a debt payment.
- **Sync**: `supplier_payments` is a new fact table, added to the sync set with one new
  `sync_enqueue_supplier_payment` function, copy-pasted from `sync_enqueue_payment`
  (`commands_v3.rs:3142`) and called inside the receive/record-payment transactions before
  commit. `operational_costs` also needs its first-ever sync wire-up (see §0 flag #4) — this
  section is the natural place to do both at once, since the auto-generated cost row needs to
  sync too for cash-flow to roll up correctly across branches.

### What it touches
`migrations/` (one new table, columns on 3 existing tables), `repo.rs` (~5 new functions
mirroring the debt pattern), `commands_v3.rs` (~4 new wrapper commands + `receive_purchase_order_v3`
extended), `sync.rs`/`commands_v3.rs` sync-enqueue wiring (2 tables), a new
`src/app/suppliers/page.tsx` (or a new tab in `inventory/page.tsx`), the PO-receive modal in
`inventory/page.tsx`, and `finance/page.tsx`'s costs tab gets richer for free via the
`reference_type` linkage (no logic change needed there, just visible new rows).

### Effort estimate
**3-4 days**, including the mandatory money-path tests (AGENTS.md R3: "any code path that
touches an amount... gets a test, no exceptions").

## 2. Per-terminal licensing

### Current state (grounding facts)
- `LicensePayload` (`license-core/signed.rs:18`) is keyed by `branch_id` + one
  `machine_fingerprint` — a license row is **already, structurally, a one-device object**
  (`Supabase license` table: `device_token UNIQUE`, one fingerprint per row). Nothing today
  actually charges "per branch regardless of terminal count" at the data-model level; the gap is
  entirely in **billing visibility and UX**, not in the crypto/activation mechanism.
- Installing on an unpaid terminal #3 today: activation already fails cleanly — the new
  machine's fingerprint matches no existing `license` row, so `accept_renewal`'s fuzzy-match
  check rejects it. The failure mode is correct; the **error message** is generic, not
  "this branch has N active terminals, contact your rep for another seat."
- **KDS is already fully exempt from license gating** — `list_kitchen_orders_v3` takes no
  `license: State<...>` parameter at all, unlike every other back-office `_v3` command. This is
  the right end-state, but it's an *absence* (nobody wired a check in), not a designed,
  documented exemption. §0 flag #2 already noted the local/cloud `terminal` table this section
  needs doesn't exist yet despite `SCHEMA_V3.md` claiming otherwise.
- Machine fingerprint (`license-core::fingerprint::MachineFingerprint { cpu_hash, disk_hash,
  mac_hash }`, 2-of-3 fuzzy match) is the correct terminal-identity primitive already and needs
  **no changes** — it was built as a device-binding lock, and a terminal *is* a device.

### Schema

```sql
-- Supabase: give the existing 1-row-per-device license table an explicit terminal type
ALTER TABLE license ADD COLUMN terminal_type TEXT NOT NULL DEFAULT 'CASHIER'
  CHECK(terminal_type IN ('CASHIER','MANAGER_STATION'));
ALTER TABLE license ADD COLUMN terminal_label TEXT;   -- admin-entered, e.g. "Cashier 2 - front counter"

-- Supabase: new table, finally delivers what SCHEMA_V3 §9.1 claimed already existed
CREATE TABLE terminal (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID NOT NULL REFERENCES tenant(id),
  branch_id           UUID NOT NULL REFERENCES branch(id),
  terminal_type       TEXT NOT NULL CHECK(terminal_type IN ('CASHIER','MANAGER_STATION','KDS')),
  license_id          UUID REFERENCES license(id),   -- NULL for KDS: free, no license row at all
  name                TEXT NOT NULL,
  machine_fingerprint JSONB,
  version              TEXT,
  last_heartbeat       TIMESTAMPTZ,
  status               TEXT NOT NULL DEFAULT 'ACTIVE' CHECK(status IN ('ACTIVE','REVOKED')),
  registered_at         TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Local SQLite `terminal` table (already specified in `SCHEMA_V3.md` §9.1, unchanged — that part of
the doc was directionally correct even though the "already aligned with an existing Prisma model"
claim was not) stays as the local fleet-visibility mirror.

### Approach

- **Cashier/manager terminals**: no change to the core activation flow. The mint form
  (`MintLicenseForm.tsx`) gains a required "اسم الجهاز" (terminal label) field and, once a branch
  has ≥1 existing license, shows a live count: "هذا الفرع لديه ٢ من ٣ أجهزة مفعّلة" (this branch
  has 2 of 3 active terminals) sourced from `SELECT COUNT(*) FROM license WHERE branch_id=? AND
  status='active'`. Minting terminal #3 is just minting another license row, now visibly grouped
  under its branch instead of being an opaque, unlabeled device fingerprint in a flat table.
- **KDS terminals**: a new, deliberately unauthenticated Tauri command
  `register_kds_terminal_v3(tenant_id, branch_id, machine_fingerprint) -> terminal_id`, called
  once on first KDS launch (idempotent — re-registers/updates heartbeat on subsequent launches).
  Writes a `terminal` row with `terminal_type='KDS'`, `license_id=NULL`. No activation key, no
  expiry, no payment — this makes the *existing* implicit "KDS just isn't checked" behavior into
  an explicit, self-documenting registration instead of an absence someone could accidentally
  "fix" by adding a license check to `list_kitchen_orders_v3` later.
- **Unpaid terminal #3, better UX**: the existing Settings → License activation failure path
  gets a specific error variant when the fingerprint-match failure happens on a branch that
  already has ≥1 active license (i.e., "this branch exists and is licensed, just not for this
  device") vs. a branch with zero licenses (unknown/new install) — the former shows "لم يتم
  العثور على ترخيص لهذا الجهاز. هذا الفرع لديه {N} ترخيص مفعّل — تواصل مع المندوب لإضافة جهاز
  جديد," the latter keeps today's generic message. This is a wording/branching change in the
  existing error path, not new cryptographic logic.
- **Billing model**: unchanged mechanically (still N separate signed license blobs), the
  contribution here is making "N" visible, labeled, and grouped by branch in the admin console —
  which is the actual thing "a branch with 2 terminals pays double" requires: not a new pricing
  engine, a way to *see and mint* the second seat instead of it being an undocumented possibility.

### What it touches
Supabase `schema.sql` (2 columns + 1 new table), `apps/control`'s licenses/mint pages (seat count
UI, terminal label field), a new unauthenticated `register_kds_terminal_v3` command in
`commands_v3.rs`, the local SQLite `terminal` table (already speced, needs an actual migration —
currently just documented, not confirmed built), and the License settings page's error-message
branching in the frontend.

### Effort estimate
**4-5 days.** No changes to the signing/verification cryptography (`cloud.rs`,
`fingerprint.rs`) — this is additive schema + admin-console UX + one new free-registration
command.

## 3. Owner dashboard

### Current state (grounding facts)
- There is **no dashboard page**. `App.tsx` hardcodes `activeView = "pos"` for every role;
  `renderContent()`'s switch only changes on manual sidebar clicks. OWNER, MANAGER, CASHIER,
  KITCHEN all land on the order-taking screen after login.
- Reports page = revenue-only (`get_sales_report_v3`: total sales, order count, top items, staff
  performance, inventory status). No cost data anywhere in it.
- Finance page = 4 independent tabs (revenue / costs / invoices / taxes), **zero cross-tab math**
  — `totalCosts` is summed locally in the costs tab and never subtracted from revenue anywhere.
  There is no profit or cash-flow figure anywhere in the app today.
- Sync set today: **orders/order_items/payments only.** Nothing about costs or suppliers reaches
  the cloud, which is the hard blocker for cross-branch comparison (§0 flag #4).
- `permissions.ts` nav confirms the role split is already correctly scoped for everything
  *except* what lands first: `CASHIER_NAV` = pos/shift/debt-readonly; `KITCHEN_NAV` = kds/shift
  only; `OWNER_NAV` is the largest nav array but `pos` is item 0 in every role's array and is
  always the initial view — nothing OWNER-specific about the landing experience today.

### Schema
No new tables. This section is a new aggregate **query** command plus a new **page**; the money
data it reads already exists once §1 lands (`supplier_payments`) and once `operational_costs`
gets its first sync wire-up (§0 flag #4, delivered as part of §1's work).

### Approach

- **One command, parameterized by the caller's own `Scope`, not two.** `Scope::Tenant` (Owner)
  vs `Scope::Branch` (Manager) already resolves to different `WHERE` predicates via the existing
  `scope_predicate` mechanism — a single `get_dashboard_summary_v3(session_token, range)` command
  aggregates `orders` (revenue), `operational_costs` + `supplier_payments` (money out), and
  `debtors.balance_cents` (outstanding debt), grouped by branch, filtered by the resolved scope.
  Owner gets every branch in the tenant; Manager gets exactly their one branch. No second command,
  no second UI component — the same dashboard page renders a 1-row or N-row branch table
  depending on who's looking at it.
- **Local-first, cloud only for what's genuinely cross-branch.** Per ARCHITECTURE_V3 §5/§7,
  local SQLite is authoritative for *that branch's own* numbers — a single-branch tenant, or an
  Owner physically at one branch's machine, gets a fully offline-capable dashboard from local
  data alone (this matters: AGENTS.md prime directive #5, "the restaurant has no internet," must
  hold for the Owner's own numbers same as everyone else's). **Cross-branch comparison for a
  multi-tenant owner requires the Supabase aggregation layer** and therefore requires internet —
  this is not a shortcut to avoid, it's the documented cloud/local split. The local dashboard
  command ships first and works standalone; a cloud-backed "all branches" view (likely surfaced
  in `apps/control` rather than crammed into the offline Tauri app, since that's already the
  back-office/aggregation surface) is the second, dependent piece.
- **Content**: KPI row (revenue / money out / net profit / outstanding debt, period selector),
  a revenue-vs-cost-out chart, a branch-comparison table (revenue/costs/profit/orders/avg ticket
  per branch, sortable — so a 10-branch owner spots the losing branch immediately), a debt widget
  (top outstanding debtors), a supplier widget (top outstanding supplier balances, from §1).
  Orders move to being a manager-level nav item, not the owner's headline — matches the request
  directly ("orders are a manager-level detail, not an owner headline").
  **Open question, not assumed**: no charting library was confirmed present in the current
  `package.json` during exploration — before implementation, confirm whether one exists
  (`recharts` or similar) or whether adding one is itself a small decision to flag to the user.
- **Role-scoped landing**: `PosLayout`'s initial `activeView` becomes role-derived instead of the
  literal string `"pos"` — OWNER/ACCOUNTANT → `"dashboard"`; MANAGER/CASHIER/KITCHEN → `"pos"`/
  `"kds"` as today (managers run the floor, they don't need a KPI screen as their first click).
  This is the one concrete wiring change implied by the request; it is **specified here, not
  implemented**, per "write no code."

### What it touches
New `src/app/dashboard/page.tsx`, new `get_dashboard_summary_v3` command in `commands_v3.rs`
(read-only aggregate, `Scope`-parameterized, needs a test per branch-scope vs tenant-scope
correctness), the sync-set widening from §1 (shared dependency, not duplicated work),
`permissions.ts` (add a `dashboard` nav entry, Owner/Manager only), `App.tsx` (role-derived
initial view), and possibly `apps/control` for the cloud-backed multi-branch view (separate,
dependent follow-on, not part of this estimate).

### Effort estimate
**5-6 days** for the local-first dashboard + role-scoped views (depends on §1 existing first for
supplier data and the sync-set widening). The cloud-backed cross-branch view in `apps/control` is
a separate, additional estimate once this lands — not included above.

## 4. Loyalty rethink

### Current state (grounding facts)
- `loyalty_cards.tier` is a real, already-migrated column (`CHECK(tier IN
  ('BRONZE','SILVER','GOLD','PLATINUM'))`) — but **nothing ever writes anything except `BRONZE`
  into it.** `TIER_CONFIG` (thresholds 0/500/1500/3000, multipliers 1/1.2/1.5/2) exists **only in
  the frontend**, computed at render time, never persisted, never read by Rust.
- **No redemption code path exists at all** — no `redeem_loyalty_points` command or repo
  function anywhere. `FEATURE_TRUTH.md` already documents this as `STUB`.
- `loyalty_transactions` (`type IN ('EARN','REDEEM','ADJUST','EXPIRE')`) is the right shape for a
  real ledger and is already a Fact table per `SCHEMA_V3.md` §6 — `REDEEM`/`ADJUST`/`EXPIRE` are
  defined and simply unused today.
- Rewards catalog is `localStorage`-only, per-device, no backing table — invisible to any other
  terminal in the same branch, let alone other branches.
- Card/phone lookup at checkout is a plain `<input>` with an `Enter`-keydown handler — this is
  already correct keyboard-wedge scanner support; **no hardware or input-handling change needed.**
- **Integrity gap** (§0 flag #5): `earn_loyalty_points_v3` takes the point amount from the
  caller rather than computing it from the order total server-side, and the POS silently
  swallows failures with an empty `catch {}`.

### Schema

```sql
CREATE TABLE loyalty_tier (
  id                TEXT PRIMARY KEY,
  tenant_id         TEXT NOT NULL REFERENCES tenant(id),
  name              TEXT NOT NULL,             -- "برونزي", "فضي", ...
  min_points        INTEGER NOT NULL,
  points_multiplier REAL NOT NULL DEFAULT 1.0,
  sort_order        INTEGER NOT NULL,
  updated_at_hlc TEXT NOT NULL, device_id TEXT NOT NULL, deleted_at TEXT, rev INTEGER NOT NULL DEFAULT 1
);

CREATE TABLE loyalty_reward (
  id                  TEXT PRIMARY KEY,
  tenant_id           TEXT NOT NULL REFERENCES tenant(id),
  name                TEXT NOT NULL,           -- "قهوة مجانية", "خصم 10%"
  points_cost         INTEGER NOT NULL,
  reward_type         TEXT NOT NULL CHECK(reward_type IN ('FREE_ITEM','DISCOUNT_FIXED','DISCOUNT_PERCENT')),
  value_minor         INTEGER,                 -- for DISCOUNT_FIXED
  value_percent_bps   INTEGER,                 -- for DISCOUNT_PERCENT
  linked_menu_item_id TEXT REFERENCES menu_item_default(id),  -- for FREE_ITEM
  is_active           INTEGER NOT NULL DEFAULT 1,
  updated_at_hlc TEXT NOT NULL, device_id TEXT NOT NULL, deleted_at TEXT, rev INTEGER NOT NULL DEFAULT 1
);
```

No new fact table — `loyalty_transactions` already has the right shape and just needs real
writers for `REDEEM`. `loyalty_cards.tier` stays as-is: a cached, derived value (same pattern as
`debtors.balance_cents`), recomputed whenever points change.

### Approach

- **Accrual moves server-side, into the payment transaction.** Instead of the POS calling
  `earn_loyalty_points_v3` separately after payment succeeds (today's silently-swallowed call),
  `finalize_order_with_payment_v3` itself computes `floor(order_total_minor / 100) *
  tier_multiplier_at_time_of_purchase` and writes the `loyalty_transactions` EARN row in the
  **same transaction as the payment**. If it fails, the payment fails — matches R5's "if the
  audit write fails, the mutation fails" philosophy extended to points as another fact of the
  sale. This closes the integrity gap in §0 flag #5 as a structural consequence, not a bolted-on
  fix.
- **Tier recompute**: a small pure function `tier_for(points, tenant's loyalty_tier rows sorted
  by min_points)` — called at the same point points change, writes the result back onto
  `loyalty_cards.tier`. Owner configures tiers once via a new panel in Settings (existing
  settings-page precedent), all branches read the same tenant-scoped rows.
- **Real redemption**: new `redeem_loyalty_reward_v3` command. Cashier looks up the card/phone
  (already works), sees available points and the active reward catalog, applies **one** reward
  to the current cart — which becomes a real order-level discount line using the *existing*
  discount mechanism (no new discount type invented), plus a `REDEEM` (negative points)
  `loyalty_transactions` row, both inside the same transaction as order finalization so
  redemption and the discount it produced are atomic and auditable together. Per AGENTS.md prime
  directive #4 ("the threat model is the employee... can a cashier steal, can the owner detect
  it"): a redemption is money leaving through a side door exactly like a manual discount, so it
  gets the same `audit_log` rigor, non-negotiably.
- **Phone-first, card-optional.** Card issuance stays as-is for customers who want a physical
  card, but `lookup_loyalty_card_v3` gains a fallback: if no card matches, resolve via
  `customer.phone → customer_id → any card issued to that customer`. For a Syrian customer who
  is more likely to remember a phone number than carry a card, this means "give your phone
  number" is a fully valid checkout flow with **zero new UI** — the phone-lookup pattern already
  exists elsewhere (`list_customers`, `get_customer_detail_v3`), this just extends it into the
  loyalty lookup path. No physical card, no app — matches the "no app required" constraint
  directly.
- **Rewards catalog moves off `localStorage`** into `loyalty_reward`, config-layer like the menu
  (`menu_item_default` pattern) — Owner-managed, all branches/terminals see the same catalog
  immediately, no more per-device drift.
- Deliberately **not** designed here, per `ARCHITECTURE_V3.md`'s explicit "notifications, promo
  suggestions... are Phase 2, don't build them from this doc": birthday rewards, referral bonuses,
  push-style re-engagement prompts. The tier/reward *mechanism* proposed above is extensible
  enough to carry those later without another schema break, but specific campaigns are out of
  scope for this task.

### What it touches
